# Implementation Plan: TradingAgents Dashboard UI

**Branch**: `250-dashboard-ui` (originally) — implementation has shipped via 17 dashboard PRs + 1 revert (PR #251) across the #249–#266 window, all merged to `main`
**Date**: 2026-05-11
**Spec**: [`spec.md`](spec.md)
**Status**: Retroactive plan — the spec was written but `/speckit.plan` was skipped at the time. This plan documents the as-built implementation, audits it against the spec, and enumerates the gaps that drive `tasks.md`.

## Summary

The TradingAgents Dashboard surfaces the multi-agent debate framework as a mobile-first web UI on the operator's existing Hetzner VPS. Engine subprocess writes `progress.json` + `events.jsonl` per FR-020/021 schemas; FastAPI dashboard reads them. Per User Story 1 (P1), the homepage shows today's ratings + paper portfolio + cost. Per User Story 4 (P3), the homepage exposes an ad-hoc trigger input. Both are partially shipped; gaps enumerated below drive `tasks.md`.

## Technical Context

**Language/Version**: Python 3.10+
**Primary Dependencies**:
- `fastapi` + `uvicorn` (dashboard backend)
- `jinja2` (server-side templates)
- HTMX 2.0.4 (CDN, polling)
- Tailwind CSS (CDN, styling)
- `langgraph` (existing 12-agent framework)
- `pydantic` v2 (FR-020/021 schemas via `tradingagents/engine/schemas.py`)

**Storage**:
- `~/.tradingagents/engine/current/{progress.json, events.jsonl}` — engine state (atomic writes per PR #249)
- `~/.tradingagents/logs/<TICKER>/TradingAgentsStrategy_logs/full_states_log_<DATE>.json` — per-ticker debate prose (~400 KB each)
- `~/.tradingagents/paper/live.json` + `live.events.jsonl` — paper portfolio (Spec 002)

**Testing**: pytest; markers `unit`, `integration`, `smoke`. Dashboard tests = 23 route (`tests/test_dashboard_app.py`) + 22 state-reader (`tests/test_dashboard_state_reader.py`) = 45 total; engine tests = 53 across 5 files. Smoke gate at `scripts/dashboard_smoke.sh` (14 ✓ assertions per local + VPS smoke output).

**Target Platform**:
- VPS: Hetzner CPX31 / Ubuntu 24.04 / Caddy ≥2.x / Podman + Quadlet / systemd ≥240
- Browser: mobile-first (any modern phone browser)

**Project Type**: web service (FastAPI server-rendered + HTMX polling)

**Performance Goals**:
- SC-008: phone cold-tap-to-rendered ≤10s (target ≤5s)
- SC-009: live-event update latency ≤5s during run
- Polling cadence: 3s active / 30–60s idle (FR-016)

**Constraints**:
- No token streaming (FR-002 — uses per-node `agent_started`/`agent_finished` events)
- Engine + dashboard subprocess-isolated; engine spawns via `systemd-run --user`
- Trigger endpoint reachable only from `127.0.0.1` (FR-010 — Caddy doesn't proxy `POST /trigger/*`)
- Read-only paper portfolio access (Constitution IV)

**Scale/Scope**: Single operator, 25-ticker default watchlist, ~$10/day LLM, ~3–4hr/day wall-clock

## Workflow Going Forward (closes G-10)

1. **One task = one PR.** No bundling unrelated tasks.
2. **No self-merging.** Operator reviews + merges every PR.
3. **No spec deviation without amendment.** If a task surfaces a need to deviate, draft an amendment first, get approval, then code.
4. **No new fallbacks.** If the existing data source is empty, render the spec-defined empty state — do not silently substitute alternative data.
5. **Smoke gate (`scripts/dashboard_smoke.sh`) must remain green** at every PR.
