# Phase 1 Data Model: Experiments Scaffolding

Formalizes the entities the spec identified, with field names, validation rules, and relationships. All "data" here is filesystem-resident — no databases.

---

## Entity: Experiment

A single test of one variation against a baseline. The unit of work for the entire scaffolding.

**Identifier**: `<YYYY-MM-DD>-<NNN>-<short-name>` where:
- `<YYYY-MM-DD>` is the calendar date the experiment was created (UTC)
- `<NNN>` is a 3-digit zero-padded sequence number, monotonically increasing within the same date
- `<short-name>` is a 2-4 word kebab-case slug derived from the hypothesis

**Examples**:
- `2026-05-02-001-mr1-contradiction`
- `2026-05-02-002-pm-blind`
- `2026-05-15-001-order-randomization`

**Validation rules**:
- Date format strict: `YYYY-MM-DD` with leading zeros (regex: `^\d{4}-\d{2}-\d{2}$`)
- Sequence: `\d{3}`, no skipping (the new-experiment command auto-finds the next free slot for the date)
- Slug: lowercase letters, digits, hyphens only. Length 1–40 chars. No leading/trailing hyphens.
- Full ID regex: `^\d{4}-\d{2}-\d{2}-\d{3}-[a-z0-9][a-z0-9-]{0,38}[a-z0-9]$`

**Storage**: `experiments/<id>/` directory at repo root.

**Required files** (created by new-experiment command):
- `HYPOTHESIS.md` (template-populated, requires manual fill)
- `PARAMS.json` (initially `{}`)
- `run.sh` AND `run.ps1` (one-line repro stubs)

**Required files** (produced during/after run):
- `results.csv` or `results.jsonl` or domain-appropriate output (gitignored)
- `ANALYSIS.md` (manually written after the run)

**Relationships**:
- 1:1 with a row in `findings.md` (aggregator pulls one summary per experiment)
- 0..N rows in any backtest CSV that carry `experiment_id == <id>`

## Entity: HYPOTHESIS file

Markdown record of what the experiment tests.

**Path**: `experiments/<id>/HYPOTHESIS.md`

**Required sections** (template-enforced):
- `# Hypothesis: <experiment short name>`
- `## What we're testing` — one paragraph; what knob varies, against what baseline
- `## Why we expect <prediction>` — one paragraph; the predicted outcome and why
- `## Success criterion` — one bullet list; what observation would confirm the prediction
- `## Source idea` — link to the entry in `docs/EXPERIMENT.md` (e.g., "MR-1", "WC-12") or `null` if novel
- `## Cost estimate` — single line; expected LLM spend (Principle III gate); empty for zero-cost experiments

**Optional sections**:
- `## Notes` — any pre-run thinking
- `## Related experiments` — link IDs of prior related experiments

**Validation**: The runner verifies HYPOTHESIS.md exists and is non-empty before starting (FR-003 enforcement). Detailed section-level validation is out of scope for v1; future work may add a `hypothesis_check` gate.

## Entity: PARAMS file

JSON record of the exact knobs varied vs the baseline.

**Path**: `experiments/<id>/PARAMS.json`

**Schema**:
```json
{
  "config_overrides": {
    "<key>": <value>,
    ...
  },
  "explicit_flags": {
    "--debate-rounds": 1,
    "--analysts": "market,news,fundamentals",
    ...
  },
  "baseline": "Anthropic Sonnet-deep + Haiku-quick + 1/1 rounds + 3 analysts (market/news/fundamentals)",
  "notes": "any free-text manual annotations"
}
```

**Auto-populated keys**:
- `config_overrides` — written by the runner from `--config-override KEY=VALUE` flags, per R-007 (`research.md`)
- `explicit_flags` — written by the runner from non-default named flags

**Manually-populated keys**:
- `baseline` — researcher describes what's being held constant
- `notes` — free-form

**Validation**:
- Must be valid JSON (runner refuses to start if not, per edge case handling)
- `config_overrides` MUST be an object (not array, not scalar)
- Empty file (`{}`) is valid (means "running the unmodified baseline as a fresh data point")

## Entity: ANALYSIS file

Markdown record of what was found and what was decided.

**Path**: `experiments/<id>/ANALYSIS.md`

**Required structure**:
```markdown
# Analysis: <experiment short name>

<one-line summary — first non-empty line after the H1>

## Result

<what the data showed>

## Decision

<what we're doing because of it: shelve idea, escalate to bigger run, change approach, etc.>
```

**Optional sections**:
- `## Detailed findings`
- `## Limitations`
- `## Next experiment`

**Summary extraction rule** (per R-001): The aggregator extracts the first non-empty line that follows the `# ` heading and appears before any other heading. Bold/italic markers around the summary are preserved in the output.

---

## Entity: findings.md

Project-root aggregation of one-line summaries from all experiments.

**Path**: `findings.md` (repo root)

**Format** (generated, do not hand-edit):
```markdown
# Findings

Living index of experiment one-line summaries. Auto-generated by `scripts/findings_aggregate.py` — do not edit by hand. To update: `python scripts/findings_aggregate.py`.

Last updated: <ISO 8601 datetime>
Total experiments: <N>
Completed: <N_with_summary>
Pending analysis: <N_without>

---

## 2026-05-02-002 — pm-blind

> When the PM is blind to the bull/bear debate, ratings are unchanged in 18 of 20 cases — the debate adds no information.

[hypothesis](experiments/2026-05-02-002-pm-blind/HYPOTHESIS.md) · [analysis](experiments/2026-05-02-002-pm-blind/ANALYSIS.md)

## 2026-05-02-001 — mr1-contradiction

*pending analysis*

[hypothesis](experiments/2026-05-02-001-mr1-contradiction/HYPOTHESIS.md)

---
```

**Ordering** (per R-002): newest experiment at the top.

**Validation**:
- Aggregator never raises on a malformed experiment dir; it includes the dir in the listing with a "summary missing" or "ANALYSIS.md missing" marker per FR-014.
- Empty `experiments/` produces a `findings.md` with the header + a "no experiments yet" placeholder, no error.

## Entity: experiment_id column on backtest CSV

A column added to the existing CSV schema produced by `scripts/backtest.py`.

**Position** (per R-004): last column.

**New schema**:
```
ticker, analysis_date, rating, error, run_seconds, deep_model, quick_model, debate_rounds, analysts, experiment_id
```

**Values**:
- Empty string `""` when `--experiment-id` not provided
- Otherwise the full Experiment ID string per its regex

**Backward compatibility**:
- Old CSVs without the column are readable: `pandas.read_csv` produces a DataFrame missing the column, which downstream code treats as `experiment_id = ""` for all rows.
- New CSVs ARE readable by the unmodified upstream `analyze_backtest.py` because the new column is at the end (no positional shift).

---

## Cross-entity invariants

1. **One ID, one directory** — each Experiment has exactly one directory at `experiments/<id>/`. The new-experiment command refuses to create a duplicate.
2. **One ID, many CSV rows** — a single Experiment may produce many backtest rows (one per (ticker, date) pair); all carry the same `experiment_id`.
3. **One directory, one summary line in findings.md** — even when ANALYSIS.md is missing, the aggregator emits a placeholder line for the directory.
4. **PARAMS.json `config_overrides` reflects runtime overrides** — when the runner uses `--config-override KEY=VALUE` and writes to PARAMS.json (per R-007), the keys/values match what was actually applied to the runtime config (not just what the user typed).

## State transitions

An Experiment passes through these states implicitly (no formal state field):

```
NEW          ← directory created by new-experiment command, HYPOTHESIS.md unfilled
PLANNED      ← HYPOTHESIS.md filled out, PARAMS.json filled out (or empty), ready to run
RUNNING      ← runner started, results.csv being appended
RAW_DONE     ← runner finished, results.csv complete, no ANALYSIS.md yet
ANALYZED     ← ANALYSIS.md exists with one-line summary present
ARCHIVED     ← long-term storage (future); not implemented in v1
```

The aggregator distinguishes RAW_DONE / ANALYZED / earlier states via the presence/contents of `ANALYSIS.md` only.
