## P1 — Latent runtime bugs

These are not style — they will or could break at runtime.

### 1. `scripts/diagnose_uw_quality.py:169` — `F821 Undefined name datetime`
```python
md.append(f"_Generated {datetime.now().isoformat()}_\n")
```
`datetime` is never imported in this file. Calling this script will raise `NameError`. The script may not be in active use (post-A1 finding), but it ships in the repo.

**Fix**: add `from datetime import datetime` at top.

### 2. `scripts/diagnose_uw_quality.py:217` — `B023 Function definition does not bind loop variable`
```python
for kw in keywords:
    ...
    mean_hits = sub["kw_hits"].apply(lambda h: h.get(kw, 0)).mean()
```
The lambda captures `kw` by reference; in some pandas execution orders this can yield wrong results (all rows compute against the LAST `kw` instead of each iteration's value).

**Fix**: `lambda h, kw=kw: h.get(kw, 0)` to bind via default arg.

### 3. `tradingagents/llm_clients/azure_client.py:50` — `F811 Redefinition of validate_model`
```python
from .validators import validate_model  # line 7
...
def validate_model(self) -> bool:  # line 50 — shadows the import
```
The instance method shadows the module-level import. Function works (the method just always returns True), but the import is dead and any other code in the file referencing the imported `validate_model` would silently get the method instead.

**Fix**: remove the unused import on line 7 (preferred) OR rename the method.

### 4. `scripts/diagnose_uw_quality.py:157` — `F841 Unused variable cols_to_compare`
Defined but never read. Dead code; harmless but suggests the file was abandoned mid-refactor.

### 5. `tradingagents/dataflows/alpha_vantage_common.py:32` — `B904 raise without from err`
Re-raising inside `except` without `raise ... from err`. Loses traceback chain. Cosmetic but is the kind of thing that hides root causes during incident response.

## P2 — Architectural debt

### A. Three implementations of forward-α math
The same `(raw - bench)/start * 100` calculation lives in 3 places with subtly different semantics:

| Location | Behavior on partial-window |
|---|---|
| `tradingagents/graph/trading_graph.py::fetch_returns` | Returns truncated α (`actual_days = min(holding_days, ...)`) — partial result |
| `tradingagents/dataflows/returns.py::alpha_from_frames` | Returns `None` if `actual < holding_days` — strict |
| `scripts/analyze_backtest.py::_compute_returns` | Returns truncated α (mirrors `fetch_returns`) |

This semantic split is the root cause of the buffer-fix-bug discovered in `claudedocs/bear-bigram-artifact-check-2026-05-04.md`. The "90d IC" was actually ~50d because `fetch_returns` returned partial α silently. The buffer fix (commit `24e009d`) papered over the symptom; the underlying duplication remains.

**Recommendation**: consolidate to `dataflows/returns.py` with explicit `strict_window: bool` parameter. Delete `_compute_returns` from analyze_backtest. Refactor `fetch_returns` to call `alpha_from_frames`. ~2-3h refactor; reduces future surprise of the same class.

### C. Hardcoded SPY benchmark at `trading_graph.py:71`
```python
spy = yf.Ticker("SPY").history(...)
```
Per `scripts/analyze_backtest.py` help: "only SPY is currently honored — fetch_returns hardcodes SPY at trading_graph.py:205". Acknowledged as a known limit.

**Fix**: thread benchmark ticker through `propagate()` → `fetch_returns(benchmark="SPY")`. Default stays SPY. ~1h.
