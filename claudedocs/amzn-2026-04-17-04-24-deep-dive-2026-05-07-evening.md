## Findings

### F-1: First documented OPERATIONAL spec 007 fire event in the corpus

AMZN-04-17 is the **first deep-dive case where Spec 007 ACTUALLY FIRED**.
Prior deep-dives (PR #39 COP+INTC, PR #43 AMD-04-17, PR #46 AMD-04-24)
all had `fired_bull = False` because pre_rating was Underweight or Hold —
the filter requires pre_rating in {Buy, Overweight} to suppress.

AMZN-04-17 has pre_rating=Overweight + bull_score=0.78 (above T=0.60) →
**filter fires + post_rating becomes Hold**. The earlier analyzer's
"100% bull-fire rate" finding from mid-flight diagnostic gets a 4th
operational data point now (was 3/3, now 4/4 if AMZN counts as bull
commit; the analyzer needs re-running to confirm).

This complements the v1.4.4 multi-mechanism-validator framing with an
**OPERATIONAL form**: not just "PM would-have-agreed but filter didn't
fire" (behavioral-additive) but "filter actually fires + PM ends up
agreeing." Both forms validate the framing.

### F-2: AMZN-04-17 prose-vs-rating divergence — Constitution VII pattern

PM rating field on 04-17 is **Hold** (post-Spec-007-fire), but the prose
EXPLICITLY recommends "Initiate AMZN at Overweight" with detailed entry
zones ($225-245, $210-220) and stop-loss ($195). Rating reflects the
filter-suppressed commit; prose reflects the LLM-debate-synthesized
multi-week thesis.

This is another instance of `memory/reference_pm_hold_with_bullish_prose.md`
— the rating is "today's commit" (Hold per the spec 007 suppression);
prose can recommend multi-week framework (Initiate at OW with tiered
entries). Filters parse rating, not prose.

Operationally: an investor following the rating literally would NOT
build an AMZN position; an investor reading the prose would build over
several weeks. This divergence is INTENTIONAL per Constitution VII and
the spec 007 design — the filter doesn't change the underlying analysis,
only the action.

### F-3: Temporal-learning arc — pre_rating shifted OW → Hold week-over-week

AMZN-04-17: pre_rating = Overweight (LLM debate synthesized bullish
position before spec 007 intervention)
AMZN-04-24: pre_rating = Hold (LLM debate synthesized cautious position
already, before any filter)

In one week, the LLM debate ensemble shifted from "Overweight" to
"Hold" — apparently absorbing the same contrarian signal that spec 007
codifies. This is METAPHORICAL learning (no model weight updates), but
mechanistically:

1. Week 1 (04-17): bull_score=0.78 → debate ensemble lands on OW → spec 007 downgrades to Hold
2. Week 2 (04-24): bull_score=0.78 (same!) but bear_score increased 0.30 → 0.40 → debate ensemble lands on Hold directly → no filter fire needed

The LLM's contextual reasoning evidently incorporated the same
contrarian information that spec 007 measures. This is striking
empirical support for the **PM-as-multi-mechanism-validator** framing
in v1.4.4 — the PM operates at the same logical level as the filter,
not just downstream of it.

### F-4: Clamp-saturation at AMZN-04-24 (5 days to earnings)

AMZN-04-24 has effective_bull=1.000 (clamp-saturated). Calculation:
0.78 × (1 + 0.5 × 0.643) = 0.78 × 1.321 = **1.030 → clamped to 1.000**.

This is the **second observed production clamp-saturation** (first was
INTC-04-17 in PR #39; PR #45 sweep refresh observed it on AMZN). 5
days-to-earnings is the most aggressive boost regime so far in the
corpus (closest-to-earnings).

Confirms spec 008 FR-018 numerical clamp behaves consistently. If pre
had been Buy/OW on AMZN-04-24, spec 008 would have fired with maximum
post-clamp effective score margin (1.000 - 0.60 = 0.40 above threshold).

### F-5: Spec 003 percentile=None on both AMZN dates

Both AMZN logs show `spec_003 percentile = None` with `baseline = none`
and `n_history = 0`. The featurizer DID compute bull_keyword_count
(76 and 75 on the two dates), but the gate had no baseline pool.

Most likely explanation: AMZN was newly-added to the SC-009 expansion
cohort and isn't yet in any sector or per-ticker history pool. Spec
003.5's sector-baseline-fallback should pick up Tech-sector aggregation,
but `gate_baseline = none` means even the sector pool was empty for
AMZN.

This is an instrumentation-gap finding worth a small followup: spec
003.5 should expose a clearer reason why baseline=none (insufficient
sector pool for THIS ticker? sector membership not recorded?). Defer
to a future spec 003.5 amendment; not v1.4.4-blocking.
